## Architecture (the big picture)

**Hybrid: static JSON for content, Supabase for dynamic/per-user data.**

- **Node.js + Express**, plain static HTML/CSS/JS frontend (single page, client-side tab switching) — **no front-end framework, no build step**. Keep it that way; it must stay deployable on shared Node hosting and cacheable for offline use.
- **Static content** lives in `/data/*.json` (rundown, visits, speakers, hotel reference, check-in template, contact). Editable by file; this is what keeps the app fast and offline-capable. The **rundown and institutional visits are shared by all delegates** (no per-group split); only hotel/check-in data is per-delegate.
- **Supabase (Postgres)** holds only the dynamic parts: `delegates`, `favourites`, `feedback`, `announcements`. See PLAN.md §5b for the schema and Row Level Security rules (delegates can read only their own row; service-role key is server-only). Auto-reminders are computed from the rundown (not stored); the bell's read/unread state is per-device `localStorage`.
- **Auth = Supabase Auth** (email + password). **No open self-registration** — delegates are pre-created by `scripts/seed-delegates.js` (admin API, autogenerated passwords); "sign up" = first login. Credentials are emailed later by `scripts/send_credentials.py` via the **Resend API**. Password reset is Supabase's built-in flow, with Resend wired as Supabase custom SMTP (sender `noreply@thecscd.org`, already configured; API key shared at the end).

## Hard constraints to respect

- **Mobile-first is the priority.** Laptop is secondary.
- **Design:** **minimalist** — follow `color.md`. Cream + near-black with lots of whitespace and hairline 1px dividers; **no heavy/offset block shadows**. Yellow `#E6EB1C` is a **restrained accent** (active nav indicator, one CTA, slim "now" marker, soft text highlight); **pink `#EA0558` is a single signal** (the unread dot / form errors) — the app must read professional/diplomatic, not "pink". Fonts: Cinzel (display, lighter weights), Cormorant Garamond (serif accents), Lato (body).
- **Secrets:** the Supabase **service-role key never reaches the browser** (server-side only); browser uses the anon key under RLS.
- **Do NOT read, open, print, or otherwise access `.env`.** It holds live secrets. Treat it as off-limits — never `cat`/Read it, never echo its contents, and never paste its values into code, logs, or chat. Refer to variables by name only (e.g. `process.env.SUPABASE_SERVICE_ROLE_KEY`). To document required vars, use `.env.example` instead. If a value is missing, ask the user rather than inspecting `.env`.
- The user controls deploy/data timing — do not start building features until the client data (delegate list, hotels, final agenda with venues, speakers, contacts) is provided.
