## Architecture (the big picture)

**Hybrid: static JSON for content, Supabase for dynamic/per-user data.**

- **Node.js + Express**, plain static HTML/CSS/JS frontend (single page, client-side tab switching) â€” **no front-end framework, no build step**. Keep it that way; it must stay deployable on shared Node hosting and cacheable for offline use.
- **Static content** lives in `/data/*.json` (rundown, visits, speakers, hotel reference, check-in template, contact). Editable by file; this is what keeps the app fast and offline-capable. The **rundown and institutional visits are shared by all delegates** (no per-group split); only hotel/check-in data is per-delegate.
- **Supabase (Postgres)** holds only the dynamic parts: `delegates`, `favourites`, `feedback`, `announcements`. See PLAN.md Â§5b for the schema and Row Level Security rules (delegates can read only their own row; service-role key is server-only). Auto-reminders are computed from the rundown (not stored); the bell's read/unread state is per-device `localStorage`.
- **Auth = Supabase Auth** (email + password). **No open self-registration** â€” delegates are pre-created by `scripts/seed-delegates.js` (admin API, autogenerated passwords); "sign up" = first login. Credentials are emailed later by `scripts/send_credentials.py` via the **Resend API**. Password reset is Supabase's built-in flow, with Resend wired as Supabase custom SMTP (sender `noreply@thecscd.org`, already configured; API key shared at the end).
